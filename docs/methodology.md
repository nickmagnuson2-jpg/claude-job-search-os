# AI Job Search System — Methodology

An open-source framework that turns Claude Code into an **end-to-end job search operating system**: from identifying target companies to accepting an offer. The system covers six capability areas:

1. **Job search operations** — pipeline tracking, outreach, networking, and weekly retrospectives keep the search moving forward
2. **Market intelligence** — industry and company research informs targeting, outreach tone, and interview prep
3. **Application materials** — tailored CVs, cover letters, and cheat sheets generated for specific roles
4. **Interview coaching & preparation** — recruiter screenings, hiring manager interviews, and one-command prep packages
5. **Voice simulation** — realistic spoken practice with transcript debrief and anti-pattern tracking
6. **Structured record-keeping** — all experience data lives in versioned markdown, git-tracked and portable

## Profile Guard

**Hard prerequisite.** Before any generative or research skill runs, Claude checks that both files exist and contain real content — not just TODO placeholders.

| File | What it must contain |
|---|---|
| `data/profile.md` | Name, contact, compensation, availability, interests |
| `data/goals.md` | Search thesis, role types, company stage, geography, compensation, current phase |

**Skills covered:** `/generate-cv`, `/research-company`, `/research-industry`, `/prep-interview`, `/cold-outreach`, `/follow-up`, `/draft-email`, `/voice-export`, `/extract-identity`, `/review-cv`, `/review-cv-deep`, `/weekly-review`, `/scan-jobs`, `/standup`

If either file is missing or all-TODOs, Claude stops immediately and tells you exactly what to do:
- Missing `profile.md` → run `/import-cv` to populate it, or create it manually
- Missing/incomplete `goals.md` → run `/setup-goals` (the fastest path)

**Bootstrap path:** `/setup-goals` reads `data/professional-identity.md`, derives industries/non-negotiables/priority stack from it, then asks only 5–6 fields it can't infer (comp, stage, geography, phase, search thesis). Result: `data/goals.md` populated in one pass.

The guard exists because generative skills produce plausible-looking but ungrounded output when personal data is missing. Failing fast with a specific fix path is better than silent degradation.

## Structured Experience Management

All professional data lives in versioned markdown files following consistent templates:

| Data type | File | What it captures |
|---|---|---|
| Projects | `data/projects/*.md` | One file per engagement: period, role, client, responsibilities, achievements, technologies, tags |
| Skills | `data/skills.md` | Grouped by category with experience levels |
| Certifications | `data/certifications.md` | Status tracking: active, expired, renewal pending |
| Companies | `data/companies.md` | Own business entities, if applicable (founding dates, purposes, outcomes) |
| Profile | `data/profile.md` | Contact, compensation, availability, interests |
| Education | `data/education.md` | Degrees, training, qualifications |
| Goals | `data/goals.md` | Search thesis, role types, company stage, geography, compensation, current phase |
| Professional identity | `data/professional-identity.md` | Identity, strengths, growth edges, values, narrative patterns (generated by `/extract-identity`) |
| Project index | `data/project-index.md` | Lightweight index for quick matching against job ads |
| Networking | `data/networking.md` | Contacts with full interaction logs (managed by `/networking`) |
| Job pipeline | `data/job-pipeline.md` | Application tracking through stages (managed by `/pipe`) |
| To-dos | `data/job-todos.md` | Action items with priority, due dates, cross-references (managed by `/todo`) |
| Daily log | `data/job-todos-daily-log.md` | Daily progress snapshots with completion rate and streaks (generated by `/checkout`) |
| Outreach log | `data/outreach-log.md` | Chronological outreach activity log (generated by outreach skills) |
| Weekly review log | `data/weekly-review-log.md` | Weekly retrospective log (generated by `/weekly-review`) |
| Notes | `data/notes.md` | General notes and unroutable inbox captures (managed by `/remember`, `/act`) |

**Important:** `data/project-background/` contains sensitive project background information (legal disputes, failures, internal assessments) for internal reference and risk assessment only. Its contents must **never** appear in CVs, resumes, cover letters, cheat sheets, or any client-facing or interview materials.

This is the **single source of truth**. Every other feature reads from here. When you finish a project, you add one markdown file and update the index — every future CV, outreach message, and coaching session automatically has the new data.

### Why markdown?

- Git-versioned, diffable, portable
- No vendor lock-in, no database, no app subscriptions
- Claude Code reads it natively — no adapters or plugins needed
- Human-readable as a fallback if you stop using the framework

## Job Search Operations

Four sub-systems keep the search running daily.

### Daily Operations

**`/standup`** — Morning briefing. Reads goals, pipeline, todos, outreach log, and networking in parallel, then outputs:
- Pipeline health snapshot (stages and staleness)
- Today's top 3 actions with cross-references to contacts and companies
- Pending outreach with follow-up sequence positions
- One suggested priority action for the day

Reading all five data files together surfaces connections that reading them individually misses — e.g., a stale pipeline entry with an upcoming follow-up that needs to be sent today.

**`/todo`** — Task manager with pipeline and contact cross-references:
- `/todo` — view active list, flagged by priority and due date
- `/todo add <task>` — add a new item
- `/todo done <item>` — mark complete and move to done section

Cross-referencing to-dos with pipeline entries and contacts adds context that makes prioritization easier — you can see that three of your open tasks are all blocked on the same company.

**`/checkout`** — End-of-day close-out. Generates today's progress snapshot (completion rate, streak, velocity trend) and appends to `data/job-todos-daily-log.md`. Surfaces tomorrow's top 3 priorities and cross-references the weekly review. Bookend to `/standup`.

### Pipeline Management

**`/pipe`** — Application tracking through stages:

```
Researching → Applied → Phone Screen → Interview → Offer → Accepted
                                                          └→ Rejected / Withdrawn
```

Features:
- Staleness alerts: entries with no update in >7 days are flagged automatically
- Per-stage action suggestions (e.g., "7 days in Phone Screen — follow up?")
- `/pipe add <company> <role>` — add a new entry
- `/pipe update <company> <stage>` — advance to the next stage
- CV Used field updated automatically when `/generate-cv` runs for a company

Staleness alerts are the most useful feature — they surface what needs attention without requiring manual review of the whole pipeline.

### Weekly Retrospectives

**`/weekly-review`** — Structured retrospective covering:
- Pipeline health by stage (entries, average age, staleness)
- Outreach response rates (sent vs. replied, by type and channel)
- Task velocity: completed this week vs. last week vs. trend
- Research activity: companies and industries researched
- Top 5 priorities for the coming week (ranked with rationale)

Logs to `data/weekly-review-log.md` (newest entries first). Tracking task velocity trend — not just snapshot counts — is a better indicator of search health than point-in-time counts.

### Execution

**`/act`** — Autonomous execution of actionable to-dos. Classifies items into two buckets before running:

- **Bucket A (executable):** Careers page checks, company research, article reads, pipeline status updates — Claude can run these in parallel
- **Bucket B (manual):** Phone calls, in-person tasks, anything requiring human action — listed but not executed

Flow: preview the action plan → user confirms → Bucket A items execute in parallel → results summarized. Also processes items from the `inbox/` capture zone (gitignored raw notes).

The Bucket A / Bucket B classification is the key design insight — not everything in a to-do list can be automated, and pre-sorting prevents false starts.

## Market Intelligence & Research

Two depth levels: industry-wide (sector map) and company-specific (deep dossier).

### Industry Research (`/research-industry`)

Five parallel agents cover:
1. Market structure and key players
2. Funding and investment trends
3. Talent demand and hiring patterns
4. Trends, regulation, and headwinds
5. Entry strategy and positioning for the candidate

Output: industry dossier at `output/<slug>/<slug>.md` — canonical file, versioned in place.

Natural handoff: run industry first to map the sector, then `/research-company` on the specific targets it identifies.

### Company Research (`/research-company`)

Five parallel agents cover:
1. Company overview — mission, product, stage, business model
2. Funding and financials — rounds, investors, runway signals
3. People and culture — founders, leadership, team signals, Glassdoor patterns
4. News and strategy — recent announcements, pivots, hiring signals
5. Competitive landscape — positioning, market share, similar companies

Output: company dossier at `output/<slug>/<slug>.md` with:
- Conversation starters calibrated to context (coffee-chat, interview, networking)
- Similar companies ranked by fit — a shortlist for parallel targeting

### Research Dossier Standards

Both skills produce dossiers with a **two-speed reading design:**

**Speed 1 — Executive Summary** (top of every dossier): thesis/verdict, opportunity rating, top 3 reasons to target, top 3 risks, recommended next action. Fits one screen. Scan in 2 minutes.

**Speed 2 — Full dossier:** Every major section opens with a bold **BLUF** (Bottom Line Up Front) sentence summarizing the takeaway. Scan all BLUFs in 60 seconds, or read any section in full for detail.

**Evidence quality rules:**

| Tier | Type | Treatment |
|---|---|---|
| A | Primary/official (SEC filings, press releases, investor pages) | Used as-is |
| B | Reputable secondary (major press, established analysts) | Used with source noted |
| C | Aggregators/crowd (Glassdoor, LinkedIn counts, Crunchbase estimates) | Flagged with caveat |

High-impact claims (market size, funding, growth, headcount, risk events) include a confidence tag: `[Confidence: High | Medium | Low, as of YYYY-MM]`.

When credible sources disagree, both values are shown with sources and marked `[Needs verification]`. Claude never silently picks one.

Trend and news claims prioritize last-12-month sources; older sources are noted with justification.

Every dossier includes a mandatory **Evidence Summary Table** and **contradiction audit** as closing sections.

**Refresh behavior:** If a dossier already exists, Claude offers "view existing" or "refresh." On refresh, a `## What Changed Since Last Update` section lists material differences from the prior version.

### Research Handoff Chain

```
/research-industry ──→ /research-company ──→ /cold-outreach or /follow-up
  (sector map)           (deep dossier)          (act on findings)
```

Each skill explicitly suggests the next step at the end of its output.

## Application Materials

### Tailored CV (`/generate-cv`)

An 11-step workflow (defined inline in `.claude/skills/generate-cv/SKILL.md`) takes a job ad and produces a tailored CV:

1. **Analyse the role** — extract requirements, technologies, seniority, market
2. **Match from data** — scan the project index for relevant experience
3. **Read project details** — pull in the 3–6 most relevant projects
4. **Consult professional-identity** — use narrative framing and reframes to inform tone
5. **Tailor the summary** — rewrite the professional summary for this specific role
6. **Order entries** — strict reverse-chronological order
7. **Adjust skill emphasis** — highlight what matches, de-emphasise what doesn't
8. **Choose language** — match the language of the job posting (or the candidate's preferred language from `data/profile.md`)
9. **Choose format** — select the appropriate market format (see [style-guidelines.md](../framework/style-guidelines.md))
10. **Output** — clean markdown at `output/<company-slug>/MMDDYY-[role-slug].md`
11. **Generate cheat sheet** — a companion quick-reference at `output/<company-slug>/MMDDYY-[role-slug]-cheatsheet.md`, with coached answers mapped to each must-have requirement

Also updates the pipeline's CV Used field for the matched company entry.

**The key insight:** instead of maintaining one generic CV that you retrofit for each application, the system generates a role-specific CV every time, drawing from complete data. A CV for a chief of staff role pulls different projects and emphasises different achievements than one for a product manager role — even though both draw from the same data.

### Cover Letter (`/cover-letter`)

A **Problem-Solution format** based on meta-analysis of 80+ cover letter studies — leads with the company's challenge, not the candidate's background:

1. **Hook** — the specific problem or challenge the company is facing (opens from their perspective, not yours)
2. **Proof** — 2–3 concrete proof points showing you've solved that exact problem before (quantified where possible)
3. **Bridge** — how your approach maps to what they need now (forward-looking, not backward-looking)
4. **Close** — clear signal of interest and a specific ask

Target length: 250–350 words. Quality gates: uniqueness test (nothing interchangeable with any other candidate), resume separation test (no bullet-point repetition — this must add context the CV can't convey), ATS keyword weaving (3–5 terms from the job description).

Output: `output/<company-slug>/MMDDYY-cover-letter.md`. Pipeline updated automatically.

### CV Quality Gates

Two review skills catch problems before the CV goes out:

**`/review-cv`** — Fast quality-gate check against the target role. Verifies keyword coverage, claim integrity, and format compliance.

**`/review-cv-deep`** — Six parallel AI reviewers each assess the CV from their perspective:
- Recruiter — does this pass the filter?
- Hiring manager — does this show depth?
- Competitor analyst — how does this compare to other candidates?
- Skeptic — what claims are unsubstantiated?
- Copy editor — grammar, clarity, tense consistency
- Source data auditor — does every claim trace back to the project files?

Produces a consolidated report with severity-rated issues and specific rewrites. Also generates "Top 10 Probing Interview Questions" — the hardest questions this specific CV would trigger — which feed directly into the voice simulation pipeline.

### CV Quality Standards

The workflow includes a mandatory pre-output checklist:
- Keyword coverage against the job posting
- Product specificity (sub-product labels, not just parent product names)
- Claim integrity against source data
- No weakness admissions or hedging language
- Concurrent engagement explanations
- Team-fit signals (collaboration references)
- Formatting, tense, and language consistency

## Outreach & Networking

### Cold Outreach (`/cold-outreach`)

Framework selection based on contact type and context:

| Framework | Best for |
|---|---|
| Persona-Based | Shared background (alumni, former employer, mutual connection) |
| 3Ps (Problem → Personal → Proposal) | Direct connection with a clear ask |
| PAS (Problem → Agitate → Solution) | Decision-maker with a specific pain point |
| BAB (Before → After → Bridge) | Transformation/outcome focus |
| AIDA (Attention → Interest → Desire → Action) | Longer-form email with multiple hooks |

**Channel limits:** 75–125 words for email; <300 characters for LinkedIn connection requests.

**Quality gate — three questions before sending:**
1. Why you? (shared context, specific reason to reach out)
2. Why now? (timing relevance)
3. Why me? (what the recipient gets from the conversation)

**Auto-logging:**
- Contact added/updated in `data/networking.md`
- Activity appended to `data/outreach-log.md`
- Follow-up to-do created in `data/job-todos.md`
- Archive saved to `output/<company-slug>/MMDDYY-cold-outreach-[contact-slug].md`

### Follow-Up Sequences (`/follow-up`)

Sequence-aware follow-ups calibrated to position in the sequence:

| Position | Timing | Tone | Strategy |
|---|---|---|---|
| 1st follow-up | 2–3 days | Gentle nudge | Brief, assume good intent, re-state ask |
| 2nd follow-up | 5–8 days | New angle | Add a piece of value (article, insight, shared connection) |
| 3rd follow-up | 10–15 days | Brief and direct | Short, direct, acknowledge silence |
| 4th follow-up | 20–28 days | Last attempt | Explicit "last note" framing |
| 5th+ follow-up | 35+ days | Break-up | Close the loop, leave door open |

Tone-matched to prior messages by reading the full message history. Every follow-up adds new value — repeating the same message verbatim is an anti-pattern.

When run with no arguments, shows a **Stale Contact Dashboard** — all contacts with pending follow-ups ordered by urgency.

Auto-logs to same files as cold outreach (networking.md, outreach-log.md, output/).

### General Email Drafts (`/draft-email`)

Auto-detects message type:
- Thank-you note (post-interview, post-call, post-introduction)
- Status update / check-in
- Introduction request (asking a contact to intro you to someone)
- Expression of interest (inbound reach to a hiring team)
- Informational interview request

Matches tone to prior messages with the same recipient. Saved to `output/<company-slug>/MMDDYY-draft-email-[recipient-slug].md` (or flat `output/MMDDYY-draft-email-[slug].md` if no company is identifiable).

### Contact Tracking (`/networking`)

- **Add contacts** — creates a structured entry with source, context, and first interaction
- **Log interactions** — appends to the contact's interaction log (call, email, coffee, event, LinkedIn)
- **View history** — shows full interaction log for a contact with suggested next action
- **Stale flag** — contacts with no interaction in 14+ days are flagged
- **Auto to-dos** — generates follow-up tasks when a contact's next-action date approaches
- **Remove contacts** — marks as inactive with a reason

Logging every interaction (even brief ones) creates a searchable contact history that informs outreach personalization — you can reference specific conversations and show continuity across months.

## Interview Coaching

Three coaching modes, each with a detailed methodology file:

### Recruiter Screening ([recruiter-screening.md](../framework/recruiter-screening.md))

Claude plays a recruiter conducting a 15–20 minute screening call. The recruiter's job is to check boxes and decide whether to forward you — your job is to survive the filter without disqualifying yourself.

Two difficulty levels:
- **Normal** — standard screening with coaching after each answer
- **Tough** — the recruiter has read the CV carefully and targets weak points. Includes compensation pushback, gap pressure tests, and direct pin-downs ("So you don't have [technology] production experience then")

Coaching happens out-of-character between exchanges. The session pauses, Claude gives specific feedback on what went wrong and provides a stronger version using your actual data, then you say "go on" to resume.

### Hiring Manager Mock Interview ([mock-interview.md](../framework/mock-interview.md))

Claude plays a technical hiring manager probing depth on architecture, decisions, and trade-offs. Questions escalate from warm-up through depth probes to pressure questions.

The hiring manager drills down: "Walk me through exactly how you implemented [specific claim from the CV]." Surface-level answers get exposed with follow-ups.

### Full Simulation ([full-simulation.md](../framework/full-simulation.md))

An uninterrupted conversation — no coaching breaks, no feedback mid-session. Tough mode is the default. This is the closest approximation to a real call.

After the conversation ends, a structured debrief covers: overall verdict, strongest/weakest answers, anti-pattern log, cheat sheet comparison (if used), confidence rating, and top 3 improvements.

### Coaching Rules (All Modes)

- **OOC pacing:** Coach never re-enters character until you say "go on" — the discussion after each answer is where the real learning happens
- **OOC safe space:** Admitting "I have no idea what that means" out-of-character is intelligence for the coach, not a performance failure
- **Real data, not generic advice:** stronger answer versions always reference your actual project files
- **No cheat sheet access:** the coach evaluates against raw data, not your prepared scripts — so flaws in the scripts get caught

## Answering Strategies

Six quick-reference technique files in [`framework/answering-strategies/`](../framework/answering-strategies/) for handling common interview pressure points. Each has a Quick Overview section for fast scanning before calls.

**Pre-interview review:** Scan `anti-patterns.md` (60 seconds) and the Quick Overview of any relevant strategy before a recruiter call.

### 1. Blank Mind Protocol ([link](../framework/answering-strategies/behavioral-questions-without-prepared-example.md))

What to do when asked "Tell me about a time..." and you can't recall a specific example.

Four escape paths:
1. **Specific example** — you do have one, it just needs a moment to surface
2. **Pattern answer** — "What I typically do in that situation is..."
3. **Honest pivot** — "I haven't faced that exact scenario, but here's the closest..."
4. **Clarifying question** — buy time and narrow the scope

### 2. Gap Reframing ([link](../framework/answering-strategies/gap-reframing.md))

How to address missing skills without disqualifying yourself.

Pattern: **Acknowledge** (don't deny) → **Pivot** (to what you do have) → **Question back** (to reveal if the gap actually matters)

### 3. Pin-Down Defense ([link](../framework/answering-strategies/pin-down-defense.md))

How to handle when the interviewer restates your gap as a closed fact: "So you haven't done X."

This is a confidence test, not fact-finding. The coached response: "Not [X] specifically, but I [adjacent strength]. [Question back that reveals if the gap matters]."

### 4. Question-Back Technique ([link](../framework/answering-strategies/question-back-technique.md))

How to turn questions around to reveal real requirements and buy thinking time. Strategic, not evasive.

### 5. Anti-Patterns Checklist ([link](../framework/answering-strategies/anti-patterns.md))

60-second pre-call negative checklist. The things NOT to do:
- Volunteering negatives unprompted
- Essay structure (verdict last instead of first)
- Over-explaining technical details to non-technical recruiters
- Confirming the interviewer's concern
- Apologising instead of contextualising

### 6. Direct Answer Structure ([link](../framework/answering-strategies/direct-answer-structure.md))

Answer first, explain second, then stop. Fixes the "essay structure" anti-pattern where the verdict comes last and the listener has already tuned out.

## Voice Simulation Pipeline

A three-step workflow for realistic spoken practice:

```
/voice-export ──→ Claude App (voice) ──→ /debrief
   generate           practise             analyse
   prompt             by speaking          transcript
```

### Step 1: Generate (`/voice-export`)

Takes a CV file and job ad URL. Produces a self-contained recruiter simulation prompt — complete with persona, question pool (drawing from the deep review if available), call flow, and all context baked in.

The prompt is completely self-contained: no file access needed in the voice app. You paste it into a voice-capable AI app (like the Claude mobile app) and start talking.

### Step 2: Practise (voice app)

You speak out loud. No typing, no pauses for coaching. As close to a real phone screen as possible.

### Step 3: Debrief (`/debrief`)

Paste the transcript back into Claude Code. The debrief skill:
- Parses the transcript into Q&A pairs
- Rates each answer (1–5) with trust/credibility impact
- Compares against coached answers from your coaching files
- Identifies every triggered anti-pattern
- Generates a recruiter assessment (checkbox match + trust/credibility)
- Logs the session to the progress tracker

## Interview Preparation

### One-Command Prep Package (`/prep-interview`)

Takes a company name (and optionally a role and context) and produces a complete prep package in one command. Three parallel agents run simultaneously:

1. **Question mapping** — 10–12 questions this company/role is likely to ask, each mapped to the closest coached answer from your coaching files. Reveals gaps where a prepared answer doesn't exist.
2. **Company context digest** — mission, key challenge, culture signals, recent news, and 3–5 conversation starters for the interview
3. **Tactics & logistics** — opening and closing strategy, 5–7 questions to ask the interviewer, logistics checklist

Output: `output/<company-slug>/MMDDYY-prep.md` — one document, everything you need. A debrief to-do is created automatically after the interview is logged.

Separating question mapping, company context, and tactics into parallel agents produces a more complete package and explicitly surfaces gaps in coached answers before the interview, not during it.

## Progress Tracking

Structured scorecards in `coaching/progress-recruiter/` and `coaching/progress-interview/` track improvement across sessions:

- **Anti-pattern frequency counts** — each pattern is tracked with occurrence count, last-seen date, and trend arrow (improving/stable/worsening)
- **Per-session logs** — specific examples, coaching notes, strongest/weakest answers, focus areas for next session
- **Session index** — table linking to each session file with date, role, mode, and confidence rating
- **Coached answers** — refined phrasings updated as sessions reveal better ways to say things

This turns interview preparation from "I feel more confident" into measurable data: anti-pattern X fired 4 times in session 1, twice in session 2, zero in session 3.

## Output File Conventions

**Company-first hierarchy:** Every named entity (company or industry) gets its own subfolder under `output/`. All files related to that entity live inside it.

```
output/
  <company-slug>/
    <slug>.md                             ← Research dossier (NO date prefix — canonical, versioned in place)
    MMDDYY-[role-slug].md                 ← Generated CV
    MMDDYY-[role-slug]-cheatsheet.md      ← Interview cheat sheet
    MMDDYY-cover-letter.md                ← Cover letter
    MMDDYY-prep.md                        ← Interview prep package
    MMDDYY-article-[title-slug].md        ← Saved article
    MMDDYY-cold-outreach-[contact-slug].md  ← Outreach archive
    MMDDYY-follow-up-[contact-slug].md      ← Follow-up archive
    MMDDYY-draft-email-[recipient-slug].md  ← Draft email archive
  MMDDYY-draft-email-[recipient-slug].md  ← Flat fallback (no company known)
```

**Naming rules:**
- Subfolder = slug (lowercase, spaces→hyphens): `impossible-foods`, `behavioral-health-tech`
- The research dossier file name matches the folder: `output/<slug>/<slug>.md`
- **No date prefix on the dossier** — it is the single canonical file for that entity, versioned in place. Attaching a date would create stale duplicates instead of one authoritative source.
- All other files inside the folder **must include a date prefix** (`MMDDYY`)

**Examples:**
```
output/impossible-foods/impossible-foods.md          ← dossier, no date
output/impossible-foods/022526-chief-of-staff.md     ← CV
output/impossible-foods/022526-chief-of-staff-cheatsheet.md
output/impossible-foods/022526-cover-letter.md
output/impossible-foods/022526-prep.md
output/impossible-foods/022526-cold-outreach-sarah-chen.md
output/behavioral-health-tech/behavioral-health-tech.md  ← industry dossier
```

The subfolder is created on first write — no pre-creation step needed.

### Deterministic Write Layer

Every data file mutation (pipeline, networking, notes, todos) goes through a dedicated Python script rather than being written inline by the LLM. The pattern: LLM classifies and decides → script executes and writes → script returns JSON → LLM checks `status` and reports.

| Script | Target file | Operations |
|--------|-------------|------------|
| `todo_write.py` | `data/job-todos.md` | add, done, clear, sync |
| `pipe_write.py` | `data/job-pipeline.md` | add, update, remove |
| `networking_write.py` | `data/networking.md` + `data/outreach-log.md` | add, log (auto-updates outreach status on reply), remove |
| `remember_apply.py` | 8 destinations (pipeline, networking, notes, profile, etc.) | route-and-write |
| `act_apply.py` | pipeline, networking, notes (via inbox routing) | pipeline-add, contact-add, notes-add |

**Why scripts instead of inline LLM writes?** Three reasons: (1) markdown table format is brittle — slight variations break parsing in subsequent skill reads; (2) write logic appears in 5+ skills — a format change would require updating every skill; (3) scripts are testable in isolation — 176 unit tests cover all mutation paths, which LLM-inline writes cannot have. The LLM focuses on judgment (what to write); the script handles mechanics (how to write it correctly every time).

### Background Automation (n8n)

Four scheduled workflows run independently of Claude sessions, keeping the inbox and cache fresh:

| Workflow | Schedule | Output |
|----------|----------|--------|
| Gmail Fetch | Every 15 min | `inbox/` markdown files from "Job Search" Gmail label |
| Standup Cache Warm | Weekdays 8am | Pre-computed JSON in `tools/.cache/` |
| Follow-up Nudge + Dossier Freshness | Daily 9am | `inbox/` nudge files if overdue contacts or stale dossiers (>30d) exist |
| Weekly Review Reminder | Friday 4pm | `inbox/YYYYMMDD-weekly-review-reminder.md` |

The nudge and reminder workflows write to `inbox/` so `/act` surfaces them automatically in the next session. Start n8n via `tools/run_n8n.bat` — sets `NODES_EXCLUDE=[]` which is required to activate the Execute Command node (blocked by default in n8n 2.x).

### Separation of Concerns

- **`framework/`** is shareable and reusable. It contains no personal data — only methodology.
- **`data/`** and **`coaching/`** are personal and gitignored. Someone forking the repo keeps the framework and populates these directories with their own experience.
- **`CLAUDE.md`** is the routing layer. It describes the repo structure, data conventions, content exclusion rules, and tells Claude which framework file to load for each task type. Without it, Claude would need to rediscover the structure every session.
- **`.claude/skills/`** define automation entry points — multi-step workflows triggered by slash commands.
- **`inbox/`** is the raw capture zone for the `/act` skill — paste or drop anything, process it later.
- **Job search operations layer** (pipeline, networking, todos, outreach) sits on top of the experience data layer and tracks what's happening with the search in real time.

## Changelog

A first-person learning log of every major capability added since the system was first built. Each entry covers what was built, why, and what was learned.

---

### Core Interview Prep System *(foundation — not built by me)*

The original system — structured experience management, three coaching modes (recruiter screening, hiring manager mock, full simulation), six answering strategy files, the voice simulation pipeline, and progress tracking scorecards — was the foundation I started with, not something I built. Everything in the rest of this changelog is what I added on top of it.

The foundation gave me: a way to store professional experience as structured markdown, a coaching loop with anti-pattern tracking, and a voice simulation pipeline for spoken practice. What it didn't have was anything for managing the search itself — no pipeline tracking, no outreach system, no research tools, no daily operations layer. That's the gap everything below fills.

---

### `/research-company` and `/research-industry` *(Feb 2026)*

**What was built:** Parallel-agent deep dossiers with evidence quality standards (Tier A/B/C source classification, confidence tags on high-impact claims, mandatory contradiction audits, two-speed reading design with Executive Summary + section BLUFs).

**Why:** Going into outreach or interviews without research context is a straight-up disadvantage. I was either under-researched (generic conversation) or over-researched (wall of notes with no structure). Neither worked.

**Learned:** The two-speed design — Executive Summary for 2-minute scan, BLUFs for 60-second section skims — makes research actually usable under time pressure. Tier A/B/C source classification prevents overconfidence in Crunchbase estimates and Glassdoor reviews, which can be wildly off. Showing contradictions explicitly (`[Needs verification]`) instead of silently picking one value turns out to matter a lot when sources disagree on headcount or funding figures.

---

### `/cold-outreach`, `/follow-up`, `/draft-email` *(Feb 2026)*

**What was built:** Outreach drafting with framework selection (Persona-Based, 3Ps, PAS, BAB, AIDA), channel limits (75–125 words email / <300 chars LinkedIn), three-question quality gate (Why you? Why now? Why me?), sequence-aware follow-up (1st through 5th+), tone-matching to prior messages, and auto-logging to networking and outreach files.

**Why:** Outreach quality directly affects response rates. I was writing from scratch each time with no system for what framework to use, when to follow up, or how to escalate tone across a sequence.

**Learned:** Sequence position (1st vs. 4th follow-up) changes the right tone and message dramatically — a gentle nudge on day 3 is completely wrong on day 25. Full message history storage enables genuine tone-matching instead of guessing. Anchoring the hook to the company's mission (not the candidate's background) makes both outreach and cover letters feel less generic.

---

### `/networking` *(Feb 2026)*

**What was built:** Contact tracker with structured entries (source, context, first interaction) and a running interaction log for every contact (calls, emails, coffee, events, LinkedIn messages). Stale-contact flagging (14+ days), auto-generated follow-up to-dos, and a Stale Contact Dashboard view.

**Why:** Relationship context was scattered across memory, chat history, and rough notes. I couldn't remember when I last talked to someone or what we discussed without digging through email.

**Learned:** Logging every interaction — even brief ones — creates a searchable contact history that actually informs outreach personalization months later. The stale-contact flag is more useful than expected: it catches relationships that are quietly atrophying before they're effectively dead.

---

### `/pipe` *(Feb 2026)*

**What was built:** Application pipeline tracking across stages (Researching → Applied → Phone Screen → Interview → Offer → Accepted/Rejected/Withdrawn), per-stage action suggestions, and staleness alerts for entries with no update in >7 days.

**Why:** Applications were falling through the cracks without a single source of truth for stage and recency. I'd submitted things and then forgotten to follow up.

**Learned:** Staleness alerts are the most useful feature — they surface what needs attention without requiring manual review of the whole pipeline. The insight isn't "here's your pipeline" but "here's what's been sitting still too long." Per-stage action suggestions (e.g., "7 days in Phone Screen — send a follow-up?") prevent the paralysis of an open pipeline with no clear next action.

---

### `/todo` *(Feb 2026)*

**What was built:** Task manager with cross-references to pipeline entries and contacts, priority and due-date tracking, daily progress log with completion rate and streak, and velocity trend across days.

**Why:** Job search to-dos were scattered across notes, chat, and memory. No single place to see what was open, no way to know if I was getting faster or slower.

**Learned:** Cross-referencing to-dos with pipeline entries and contacts adds context that makes prioritization easier — you can see that three open tasks are all blocked on the same company. Tracking velocity trend (not just completion count) is a better indicator of search momentum than a snapshot.

---

### `/act` *(Feb 2026)*

**What was built:** Autonomous execution of actionable to-dos, with Bucket A (executable: careers page checks, article reads, company research) vs. Bucket B (manual: phone calls, in-person tasks) classification before running. Processes `inbox/` raw capture zone. Preview-then-confirm flow.

**Why:** Some to-dos are purely mechanical — checking if a job is still posted, reading a linked article, pulling company data. Running these manually was wasted time.

**Learned:** The Bucket A / Bucket B classification is the key design insight. Not everything in a to-do list can be automated, and pre-sorting prevents the frustration of Claude starting to "execute" a task that turns out to require a phone call. The preview step before execution also matters — it surfaces the action plan before committing, which catches misclassifications.

---

### `/prep-interview` *(Feb 2026)*

**What was built:** One-command interview prep package using three parallel agents: question mapping (10–12 likely questions → coached answers), company context digest (mission, challenge, culture signals, conversation starters), and tactics & logistics (opening/closing strategy, 5–7 questions to ask, logistics checklist).

**Why:** Interview prep was being done ad hoc — sometimes thorough, sometimes rushed, always starting from scratch.

**Learned:** Separating question mapping, company context, and tactics into parallel agents produces a more complete package and explicitly reveals when coached answers have gaps before the interview, not during it. The question-mapping agent checking coached answers is the most valuable piece — it turns "I think I'm prepared" into "here are the 3 questions where I don't have a strong answer yet."

---

### `/weekly-review` *(Feb 2026)*

**What was built:** Structured retrospective covering pipeline health by stage, outreach response rates (sent vs. replied), task velocity trend, research activity, and Top 5 priorities for the coming week. Logs to `data/weekly-review-log.md`.

**Why:** I needed a systematic way to assess search momentum — not just "how do I feel about this week" but actual metrics.

**Learned:** Tracking task velocity trend (not just completion count) is a better indicator of search health than snapshot counts. A week where I completed 12 tasks but velocity is declining is worse than a week where I completed 8 but velocity is up. The Top 5 priorities at the end of the review — ranked with rationale — is the most immediately useful output.

---

### Output Hierarchy Migration *(2026-02-25)*

**What was built:** Company-first `output/<slug>/` structure replacing flat `output/` files. Every named entity (company or industry) gets its own subfolder. All dated artifacts live inside the subfolder. The research dossier is the one exception — no date prefix, canonical file versioned in place.

**Why:** The flat `output/` directory became hard to navigate as files multiplied. Everything mixed together — CVs, dossiers, prep docs, outreach archives — with no grouping by entity.

**Learned:** The research dossier as a canonical no-date file (`<slug>.md`) works better as a versioned-in-place document — attaching a date creates stale duplicates instead of a single source of truth. Grouping all artifacts by company makes it fast to find everything related to one target without sorting through a flat list.

---

### `/standup` *(2026-02-24)*

**What was built:** Morning briefing reading five data files in parallel (goals, pipeline, todos, outreach log, networking) and outputting a daily brief: pipeline health, today's top 3 actions with cross-references, pending outreach with sequence positions, and one suggested priority action.

**Why:** Starting the day without a clear picture of priorities wastes time. I was opening multiple files manually and then mentally synthesizing them — that synthesis work belongs to Claude.

**Learned:** Reading all five data sources together surfaces connections that reading them individually misses — a stale pipeline entry with a follow-up due today, a contact who needs a reply before a meeting later this week. The parallel-read approach also means the briefing arrives fast enough to actually replace the manual morning scan.

---

### `/setup-goals` *(2026-02-24)*

**What was built:** Identity-aware goals bootstrapper. Reads `data/professional-identity.md`, derives industries/non-negotiables/priority stack from existing identity data, then asks only the 5–6 fields it can't infer (compensation, company stage, geography, current phase, search thesis). Writes `data/goals.md`.

**Why:** `goals.md` was being filled inconsistently or not at all. The template had too many fields, and without a guided process it was easy to skip.

**Learned:** Deriving what it can from `professional-identity.md` first — and showing the user what it derived before asking questions — dramatically reduces setup friction. When the system says "I've inferred that you're targeting growth-stage companies in health tech based on your identity doc — is that right?" it's far easier to confirm or correct than to fill in a blank field from scratch.

---

### `/cover-letter` *(2026-02-25)*

**What was built:** 3-paragraph cover letter generator with a fixed structure: hook anchored to company mission → value bridge with 2–3 proof points → close with ask. Reads profile, goals, professional identity, and relevant projects. Quality gates on specificity and length. Saves to `output/<company-slug>/MMDDYY-cover-letter.md`.

**Why:** Cover letters were being written ad hoc with inconsistent structure. Some led with the candidate's background; some were too long; none had a consistent hook strategy.

**Learned:** Anchoring the hook to the company's mission or a specific challenge they face — rather than "I am excited to apply..." — makes letters feel less generic and more like the candidate did their homework. The hook → value bridge → close structure is reliably effective once it becomes a fixed constraint rather than a suggestion.

---

### Profile Guard *(2026-02-25)*

**What was built:** Hard prerequisite checking both `data/profile.md` and `data/goals.md` before any generative or research skill runs. If either file is missing or all-TODOs, Claude stops and provides a specific fix path.

**Why:** Generative skills were producing plausible-looking but ungrounded output when personal data was missing. A CV generated without profile or goals data might look professional but wouldn't be tailored to the actual candidate or search context.

**Learned:** Failing fast with a specific fix path (run `/setup-goals`, run `/import-cv`) is better than producing plausible-looking but ungrounded output. Users don't always notice that generic output is wrong until they've already used it. A hard stop with a clear instruction is friendlier than a silent failure.

---

### `/remember` *(Feb 2026)*

**What was built:** Mid-session note capture with routing classification. Takes a note from the current conversation and routes it to the right file: contact note (networking.md), company note (output dossier), pipeline update (job-pipeline.md), profile update (profile.md), or general note (notes.md).

**Why:** Important context was being captured as raw chat text and then lost between sessions. Notes about contacts, companies, and decisions were disappearing into conversation history.

**Learned:** Routing classification needs to be explicit. Without it, everything ends up in a general notes dump that quickly becomes unsearchable. The five-category routing (contact / company / pipeline / profile / general) covers the vast majority of what gets captured mid-session.
